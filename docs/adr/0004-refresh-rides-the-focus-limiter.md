# Refresh ordering rides the focus limiter, not a dedicated queue

`r` (refresh the selected/open PR) and `R` (refresh every visible PR) re-fetch a PR's enrichment as one `Cmd::RefreshPr` per PR — review-status, threads, reviews, and (for `r`) files, all concurrently; checks fall out of the `ReviewStatusArrived -> maybe_fetch_checks` cascade, the same path initial enrichment uses. The refreshes are dispatched straight to the runtime. No second scheduler sits between `update` and the transport: the `NetworkLimiter` (ADR 0003) is the only thing that orders and bounds the work.

That works because the limiter already provides the two guarantees a refresh needs. **The one that matters** — the PR the user is looking at refreshes first — is the limiter's focus promotion: the selected/open PR is the focused PR, so its `RefreshPr` sub-fetches are interactive-effective and jump the queue. **The tier nicety** — `me-blocking` PRs refresh ahead of `waiting-on-author` ones — falls out of dispatch order: `R` sorts its refreshes by cached smart-status tier before dispatching, and the limiter's background lane is strict FIFO by insertion, so they drain in tier order. The model keeps only a `refreshing: HashSet<PrKey>` (the row indicator + the dedupe guard — a re-press while in flight is a no-op) and a completion counter (sizing the "Refreshed N" summary once the set drains).

## Considered options

- **A dedicated refresh priority queue (a `BinaryHeap` keyed by tier, a pump bounding concurrent PR refreshes to a small cap)** — built first (M11), then removed. The cap existed only to make tier order _observable_: if every `RefreshPr` were dumped on the focus-ordered limiter at once, tier order would be lost. But dispatching `R` in tier order makes the limiter's existing FIFO background lane realize the same order with no cap, no heap, no pump, and no second lifecycle to keep in sync with the limiter. The queue was ~365 lines (with its lazy-deletion heap and seq-invalidation) solving a problem the limiter already solved one layer down.
- **Teach the limiter to order by `(focus, then tier)`** — the tiebreaker ADR 0003 anticipated. Rejected: it forces each PR's tier into the transport layer, which today carries only the PR key and _derives_ focus. Initial enrichment has no tier yet (tier is computed from the very data enrichment fetches), so the field would be absent exactly when most fetches are dispatched; and a tier baked in at dispatch is the stale-priority smell ADR 0003 rejected for the dispatch-time interactive lane. Dispatch order gets the same observable result for free, and focus promotion still re-ranks live for the case that actually changes mid-batch (the user navigating).
- **Keep the two-tone glyph (dim "queued" vs cyan "refreshing")** — dropped with the cap. Without a cap there is no "queued" phase a refresh sits in; every dispatched refresh is in flight (its sub-fetches may wait on a limiter permit, but that is the limiter's `waiting` count, not a refresh phase). The indicator collapses to one "refreshing" glyph.

## Consequences

- Checks have a single owner again. `Cmd::RefreshPr` no longer fetches checks itself; a refresh evicts the PR's cached `(repo, head SHA)` checks entry so `maybe_fetch_checks` re-fetches them on the review-status arrival — even when the head SHA is unchanged (CI re-ran on the same commit, the prime "did it pass yet?" case). The previous design fetched checks in both `RefreshPr` and the reducer, double-fetching whenever the head SHA changed.
- `RefreshComplete` fires when review-status/threads/reviews/files settle; the cascade-dispatched checks fetch trails it, exactly as on initial load. So the row's checks briefly blank then refill on refresh — consistent with first load, not a new pattern.
- There is no longer a bound on _how many PRs_ refresh at once; a large `R` dispatches one `RefreshPr` per visible PR. In-flight HTTP is still bounded by the limiter (16 total / 8 background), which simply queues the rest — the same backpressure every other fan-out rides.
- The include-files union the queue did (a `RefreshPr` enqueued without files, re-pressed with files, gained files) is gone: a re-press while in flight is a plain dedupe no-op. Re-pressing after completion fetches files normally.
