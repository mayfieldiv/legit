# Network limiter is a focus-promoting priority scheduler

The shared `NetworkLimiter` was a single fair (FIFO) semaphore of 8 slots, so a fetch the user is actively waiting on — the detail body on drill-in, the selected PR's files — parked behind the entire list-wide enrichment fan-out and only loaded once that backlog drained. We replaced it with an explicit priority scheduler: pending requests are held as data in a queue, and a pump grants slots by effective lane. A hard total cap of 16 bounds all in-flight requests; a background sub-cap of 8 bounds speculative, list-wide work (the open-PR listing and the enrichment fan-out). A request's effective lane is computed by the pump, not fixed at dispatch: it is `Interactive` if it asked for that lane outright (detail body, selected files) **or** its PR is the one the user is currently focused on (the open detail PR, else the selected list PR); otherwise `Background`. Interactive-effective waiters draw against the full 16; since background can never exceed its sub-cap, at least 8 slots are always free for interactive work, guaranteed immediately and borrowing up to 16 when background is idle. Borrowing is asymmetric — background never reaches into interactive's headroom.

The scheduler (not a semaphore) is what makes **focus promotion** possible: when focus moves to a PR, `set_focus` re-ranks the queue so that PR's pending threads/reviews/checks become interactive-effective, and any the background sub-cap was holding back are granted at once on a borrowed slot. The runtime pushes the focused PR key to the limiter after each batch of messages. Moving focus away re-ranks with no extra bookkeeping. Only _pending_ requests promote — an in-flight one already holds its slot.

## Considered options

- **Keep the dual-semaphore (two `Semaphore`s, background acquires sub-cap then total)** — simpler, and it delivers the 16/8 borrow guarantee. Rejected once promotion was a requirement: a request parked on a semaphore is a blocked future you can't reach in and reclassify, so the selected PR's already-dispatched enrichment could never jump the queue. Promotion forces an explicit, inspectable queue.
- **Priority queue-jump within one 8-slot pool** — interactive waiters ahead of background but still waiting for an in-flight slot. Rejected: leaves up to one request-RTT of artificial latency on the action the user is blocked on.
- **Bypass the limiter for interactive** — truly instant, no permit. Rejected: `FetchFiles` fires per cursor move and is never cancelled, so holding an arrow key would fan out an _uncapped_ burst (the shape GitHub's abuse detection watches for), and bypassed requests wouldn't show in `NetworkStats`.
- **Static reserved lane (e.g. background 6 / interactive 2)** — permanently cuts background throughput even when idle. Rejected: borrowing gives interactive the same guarantee without taxing initial enrichment.

## Consequences

- The "8" was never a real GitHub ceiling — a comment cited a non-existent PRD (the TS reference used 10). GitHub's documented secondary-rate-limit ceiling is ~100 concurrent requests across REST + GraphQL, so a 16 total cap is comfortably safe. The `runtime.rs` constants and comment were rewritten to state this.
- This is the **Priority Queue** that `CONTEXT.md` describes, finally ported — but keyed on _user focus_ (interactive vs background), not the smart-status tier the original concept envisioned. A future tier-based ordering would slot in as the tiebreaker among background-effective waiters.
- The limiter holds a mutex over its queue; all critical sections are synchronous (no `.await` under the lock) and grants are delivered after the lock is released, so a send to a cancelled receiver can't deadlock against it.
- Checks can't fully jump the queue: the head SHA they need comes from `FetchReviewStatus`, which is one batched request for all PRs in a repo, not per-PR. A focused PR's checks still wait on that batch (dispatched first in the fan-out). `FetchChecks` carries the PR key purely so the limiter can promote it once dispatched.
- Fast list navigation fires a `FetchFiles` per PR passed, so it can momentarily drive interactive to 16 concurrent fetches and pause enrichment until the scroll settles. We deliberately **do not** cancel or debounce these: `FetchFiles` is idempotent (a PR already in `enrichment.files` is never re-fetched), so passing over a PR warms its files and they are already filled in when the user scrolls back — cancelling would discard exactly the work they are about to want. The burst is bounded by the 16 cap and pre-dates this change (it was capped at 8 before).
